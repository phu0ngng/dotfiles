## MR & Diff Hygiene Rules
- **Minimize Diffs**: Only modify lines directly related to the current task. Do not perform "drive-by" refactoring or reformatting of adjacent code.
- **Style Preservation**: Strictly adhere to the existing file's indentation, quote style, and naming conventions, even if they differ from your defaults.
- **Atomic Implementation**: Break large tasks into small, logical steps. Before writing code, use `/plan` to list exactly which files and functions will be touched.
- **No Unsolicited Refactors**: When refactoring is requested, preserve external behavior exactly and do not add new features or "improvements".
- **Verification**: Always run `git diff` and local tests before marking a task as complete.

# MR Description Guidelines
- Always format MR descriptions in Markdown.

# Git Commit Guidelines
- Use only single-line commit messages.
- Always use `git commit -s` to automatically attach the sign-off. Never manually append the sign-off line.
- Never include "Generated by Claude" or any other AI attribution.
- Never set yourself as the commit author or committer. Commits must use the user's git identity (`Phuong Nguyen <phuonguyen@nvidia.com>`). Do not override `GIT_AUTHOR_*` or `GIT_COMMITTER_*` env vars.
- When committing a fix for MR/PR reviewer feedback, the commit message must describe the fix content itself, not the fact that it addresses feedback. Avoid wording like "address reviewer feedback", "apply review comments", or "fix per review".

# Code Comments
- Keep docstring/comments short and concise. Don't expose impl details or intermediate steps/ops
  info. Don't mention anything about sprint files or optimization orders etc. Only write info that
  users or other dev needs to know to understand the code.

# Response Style
- Be terse. No trailing "summary of what I did" sections — the diff and tool output already show it.
- State results and decisions directly; skip narration of internal deliberation.
- For exploratory questions, give a 2-3 sentence recommendation with the main tradeoff, not a plan doc.

# Subagent / Parallel Work
- When writing kickoff prompts for subagents, do NOT hardcode baseline performance numbers — instruct the agent to measure its own baseline before changing code, so comparisons stay valid against the agent's actual build.
- Always specify the exact build/iteration flags the agent must use (e.g. fast-build) so iteration time stays bounded.
- Prefer reusing existing git worktrees (reset + new branch) over creating new ones for similar experiments.

# Action Safety
- Never run destructive git ops (`reset --hard`, `push --force`, branch -D) without explicit confirmation, even in auto mode.
- Investigate unfamiliar files/branches before deleting — they may be in-progress work from another agent or session.
